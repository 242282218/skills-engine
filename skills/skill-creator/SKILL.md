# 技能创建指南 (Skill Creator)

该技能提供有关如何创建有效技能的指导。

## 关于技能 (About Skills)

技能是模块化、独立的文件夹，通过提供专业知识、工作流和工具来扩展 AI Agent 的能力。我们可以将其比作针对特定领域或任务的“入职指南”——它们将一个通用的 Agent 转变为一个配备了任何大模型都无法完全具备的具体过程知识的专用 Agent。

### 技能能够提供什么

1. **特定的工作流** - 针对特定领域的多步流程或规则
2. **工具集成** - 处理特定文件格式或 API 操作的指令
3. **领域专家知识** - 公司专有的知识、业务逻辑或 Schema
4. **打包的资源** - 用于复杂且重复性任务的脚本、参考资料和模版

## 核心原则

### 简洁是关键

上下文窗口是公共资源。技能与 Agent 需要的所有其他内容（系统提示词、对话历史、其他技能的元数据以及用户的实际请求）共享上下文窗口。

**默认假设：Agent 已经非常聪明。** 只需添加 Agent 尚不具备的上下文。对每一段信息提出质疑：“Agent 真的需要这个解释吗？”以及“这一段值得消耗相应的 token 成本吗？”

优先使用简洁的示例，而不是冗长的解释。

### 设定适当的自由度

根据任务的脆弱性和变化程度，为 Agent 匹配适当的约束等级：

- **高自由度（基于文本的指令）**：当多种方法均有效、决策依赖于上下文、或有启发式方法指导时使用。
- **中等自由度（伪代码或带参数的脚本）**：当存在首选模式、允许部分变体、或配置会影响行为时使用。
- **低自由度（特定的脚本、较少的参数）**：当操作脆弱且易错、一致性至关重要、或必须遵循特定的执行顺序时使用。

你可以将执行过程想象成探险：走在悬崖边狭窄的桥上需要特定的护栏（低自由度），而在开阔的原野上则允许有多种路线（高自由度）。

### 技能的结构 (Anatomy of a Skill)

每一个技能都由一个必需的 `SKILL.md` 文件和可选的打包资源组成：

```text
skill-name/
├── SKILL.md (必需)
│   ├── YAML 前言元数据 (必需)
│   │   ├── name: (必需)
│   │   └── description: (必需)
│   └── Markdown 指令主体 (必需)
└── 捆绑资源 (可选)
    ├── scripts/          - 可执行代码 (Python/Bash等)
    ├── references/       - 打算在需要时才被加载到上下文的文档
    └── assets/           - 用于任务输出的文件（模板、图标、字体等）
```

#### SKILL.md (必需)

每一个 `SKILL.md` 包含：

- **Frontmatter (YAML 前言)**: 包含 `name` 和 `description` 字段。**这是 Agent 用来决定何时使用该技能的唯一字段**，因此务必在这里清晰全面地描述该技能是什么，以及何时应该使用它。
- **Body (Markdown 主体)**: 使用该技能的说明和指导。仅在该技能被触发后才会加载到上下文中。

#### 捆绑的可选资源 (Bundled Resources)

##### 1. 脚本 (`scripts/`)
可执行代码 (Python/Bash 等)，用于需要确定性可靠性或需要反复重写的任务。
- **何时包含**：当相同的代码面临被反复重写时，或需要确定性执行结果时。
- **示例**：`scripts/rotate_pdf.py` 用于处理旋转 PDF。
- **优势**：节省上下文 token，保证确定性，无需将脚本源码加载到上下文中即可通过命令行执行。

##### 2. 参考资料 (`references/`)
打算在此技能执行过程中，当 Agent 需要时再去主动加载到上下文中阅读的文档材料。
- **何时包含**：针对特定子领域或复杂场景才用得到的深层知识。
- **示例**：`references/finance_schema.md`、`references/api_docs.md`。
- **最佳实践**：如果文件非常大（>1万字），请在 `SKILL.md` 中说明可以使用 grep 或提取关键内容的方案。
- **避免重复**：信息应存在于 `SKILL.md` 或 references 中，不要两者同时写。在 `SKILL.md` 仅保留必不可少的流程和导航，将复杂的代码样例或长文本移入 references，避免主干文件过于臃肿。

##### 3. 资产 (`assets/`)
并非供 Agent 阅读，而是需要 Agent 原样输出、复制或用作基础脚手架的静态文件。
- **何时包含**：当技能需要为最终交付物提供固定文件时。
- **示例**：`assets/logo.png` 或 `assets/frontend-template/`。
- **优势**：将输出资源与规则文档分离。

#### ❌ 不应该包含在其中的内容

技能应仅包含 Agent 工作必需的信息。请勿创建面向人类或无关的过程性文档：
- README.md / INSTALLATION_GUIDE.md / CHANGELOG.md 等

开发过程说明、测试教程等只会增加迷惑性并浪费 Agent 的算力去分析。

### *渐进式信息披露设计原则 (Progressive Disclosure)*

技能应采用三级加载系统来高效管理 Agent 的运行内存（上下文）：

1. **元数据 (name + description)** - 常驻在环境内用于触发匹配（约100字）
2. **SKILL.md 主体** - 只要技能触发就会完整读取（控制在 500 行以内）
3. **打包资源** - 仅由 Agent 在需要时执行工具命令来按需加载解析

#### 渐进式披露模式示例

当技能支持多种变体、框架或选项时，仅在 `SKILL.md` 内部保留核心工作流。将具体变体的细节移至参考文件中：

**模式 1：包含参考资料的高级指南**
```markdown
# PDF 处理指南
## 快速入门
[基础用法与指令...]

## 高阶特性按需查阅
- **表单填写**：请主动查阅 [FORMS.md](references/FORMS.md)
- **API 参考**：详细端点请查阅 [REFERENCE.md](references/REFERENCE.md)
```

**模式 2：特定领域的组织结构**
应对跨模块的需求，可按领域拆开存储：
```text
bigquery-skill/
├── SKILL.md (全局导航与通用查询原则)
└── references/
    ├── finance.md (财务报表查询细节)
    └── sales.md (销售漏斗数据分析逻辑)
```
如此一来，当用户仅关心销售数据时，Agent 就不用阅读大量多余的财务报表逻辑。

**重要规则：**
- **避免嵌套参考链**：参考资料的层级从 `SKILL.md` 到目标文件应保持在一层以内。所有参考文件应当在主 `SKILL.md` 直达。
- **为长文件写下太长不看 (TL;DR)**：对于特别篇幅长的参考文件，在顶部写下表格目录或内容范围概述。

## 技能创建流程

按以下步骤创建一个优质技能：

1. **基于具体示例理解技能**：不要凭空捏造。弄清楚自己平时处理这类任务时用到的具体步骤是什么。
2. **规划可重用的技能内容**：如果有些样板代码每次都要用，把它提取成脚本或资源放进 `assets`。
3. **初始化技能**：在 `skills/` 下新建与技能名同名的文件夹，并包含 `SKILL.md` 入口文件。
4. **编辑技能主干**：
   - 填写 YAML Frontmatter。**再次强调：所有的“何时触发这个技能”的信息必须、也只能写在 Frontmatter 的 description 字段里面**。
   - 整理执行工作流并在 Body 里用 Markdown 表达，必须使用祈使句和动宾短语进行果断引导。
5. **不断迭代验证**：在自己真实的项目里面使用，如果发现 Agent “走弯路”或者犯错，那就在 `SKILL.md` 里补充“防掉坑指南”或修改引导词。

### 技能的命名 (Naming)
- 仅使用全小写字母、数字和连字符 (`hyphen-case`)。例如由 "Plan Mode" 写为 `plan-mode`。
- 将长度限制在 64 个字符以内。
- 倾向于使用能描述“做什么工作”的短语。
- 当能避免混淆时，可以用工具名作前缀，例如 `gh-address-comments` 优于 `address-comments`。
- 技能的根文件夹名称**必须**与 `name` 严格保持相同！
